### Relevant Learnings

#### 1. Go cross-platform CI pitfalls — build tags, filepath.IsAbs, and path-separator comparisons

- **File**: `/Users/justinbeaudry/Projects/aienvs/docs/solutions/best-practices/go-windows-cross-platform-pitfalls-2026-04-24.md`
- **Module**: `internal/fsroot`, `internal/cache`, `internal/workspace`
- **Severity**: high
- **Relevance**: `internal/adapter/contract` adds a new Go package. Any test file that calls stdio primitives (`os.Pipe`, `os.Stdin`), exercises framing logic with byte buffers, or uses `syscall`-adjacent APIs needs to be checked for build-tag discipline before the Windows CI matrix runs.
- **Key Insight**: `t.Skip("windows")` does not protect a file from compile-time symbol resolution. If any import in the file is Windows-unavailable, the file must be split with `//go:build !windows` — not guarded with a runtime skip. Also: any test assertion that compares a path string produced by `filepath.Join` against a forward-slash constant will silently fail on Windows. Use `filepath.ToSlash` on the joined side before comparing.
- **FLAG — likely relevant to this PR**: The Content-Length framing code works with byte offsets and `\r\n` CRLF line endings (LSP spec requires CRLF). If any test hard-codes `\r\n` as a Go string literal and compares against output produced by a buffer writer, the comparison is fine — but if any test reads back bytes through a platform-buffered reader that translates line endings (e.g., text-mode file I/O on Windows), framing byte counts will silently mismatch. Prefer raw `[]byte` comparisons or explicitly open pipes in binary mode.

#### 2. Reconciling spec-vs-impl drift caught at PR review

- **File**: `/Users/justinbeaudry/Projects/aienvs/docs/solutions/workflow-issues/spec-impl-drift-at-pr-review-2026-04-25.md`
- **Module**: `internal/ir`
- **Severity**: medium
- **Relevance**: This PR includes a "hand-rolled schema-vs-types parity test" — exactly the fixture-test pattern recommended in this learning's Prevention section. That parity test is the right call. The risk is that the JSON Schema files (likely in `embed.FS`) were authored before or alongside the Go types, and the two can drift on details (field names, required arrays, enum values, `additionalProperties` flags) the same way `ir-v1.md` drifted from `decode.go`.
- **Key Insight**: The parity test is only as strong as what it asserts. A test that merely checks "the schema parses without error" or "the Go struct round-trips through JSON" does not catch field-name drift or missing `required` entries. The test should assert each JSON Schema field name against the corresponding Go struct tag, and each `required` array entry against fields without `omitempty`. If the test skips those assertions, it provides false confidence — the same failure mode as the skipped duplicate-id test in Unit 7 that made an invariant invisible.
- **FLAG — directly applicable**: The PR description mentions a "spec-vs-types parity test." Before this PR merges, confirm the test exercises: (a) every field in the JSON Schema appears in the Go struct with a matching `json:` tag, (b) every `required` schema entry corresponds to a non-pointer, non-`omitempty` Go field, and (c) the JSON-RPC `method` string values in the schema match the Go constants exactly (case-sensitive). If any of these are left to prose, they are invisible to CI.

### Recommendations

1. **Audit framing tests for Windows line-ending hazards.** The LSP Content-Length header uses `\r\n`. Any test that writes a frame and reads it back through an `os.Pipe` or `bufio.Reader` should use raw byte comparison, not string comparison, to avoid platform text-mode translation. Run `GOOS=windows GOARCH=amd64 go vet ./...` locally before pushing — it is cheap and catches Rule 1 compile failures before CI.

2. **Strengthen the parity test beyond round-trip.** Do not let the schema-vs-types test pass on "no decode error." Add explicit assertions: Go struct tag == JSON Schema property name, `required` entries == non-optional fields, method string constants == schema enum values. A weak parity test is the exact failure mode documented for Unit 7's skipped invariant test.

3. **Use `//go:build !windows` for any test importing platform-specific packages.** This package introduces stdio framing; if any test uses `syscall` or `golang.org/x/sys/unix` for pipe setup, it must be in a `_unix_test.go` file with the build tag — not guarded by `runtime.GOOS` at runtime.

4. **embed.FS path separators are always forward slashes.** `embed.FS` normalizes paths to `/` regardless of OS — this is safe and correct. No separator normalization is needed for embedded schema file reads. Do not apply `filepath.FromSlash` to paths passed to `fs.ReadFile` on an `embed.FS`; it will break on Windows.

5. **Expect 1-2 auto-reviewer fix rounds.** Per the PR review cadence memory, gemini + copilot + codex all auto-review on push. If schema field names or method string constants diverge even slightly from the JSON Schema content, expect conflicting reviewer threads. The reconciliation procedure from Learning 2 applies: identify which side carries the mechanical invariant (the Go constant or the embedded schema), and move the prose, not the enforcement point.</result>

**1. Is the error surface machine-actionable?**

Yes for the six defined ErrorClass constants — they map cleanly to retry (adapter-timeout), abort-and-report (adapter-panic), abort-and-reconfigure (adapter-protocol-mismatch, adapter-capability-lied), and abort-and-fix-trust (adapter-exec-denied, adapter-undeclared-output). **2. Are OpWarning fields structured for programmatic consumption?**

Partially. `concept_id` (stable string key) and `status` (closed enum `degraded|partial`) give an agent a usable two-field branch surface. The `note` field is prose-only. The design is usable as-is if the PR 3 spec documents that `concept_id + status` is the complete stable branch surface and that `note` MUST NOT be parsed programmatically. If there are v1 warning scenarios where two different reasons produce the same `status` on the same `concept_id`, a `code` sub-classifier field will be needed. This is W1.

**3. Can an adapter author understand how to implement an adapter from this package alone?**

Mostly yes. The package doc in `/Users/justinbeaudry/Projects/aienvs/internal/adapter/contract/framing.go` (line 1-9) explicitly says the spec doc ships in PR 3 and points there. The JSON Schemas give a formal field contract, and the parity test validates alignment. The one genuine confusion point is O2 — `DeclaredOutput.JSONPath` and `ToolOwnedKindJSONPointer` refer to two different locator standards, and the test fixtures use both inconsistently. An implementer of `write_tool_owned` will be confused about which standard to use in the `declared_outputs` list vs. the op locator field.

**4. Does the contract leak user-facing concerns into the wire layer?**

No. The package is byte-pure. There are no ANSI codes, no color references, no terminal-width logic, and no formatting. `Error.Message` is a plain string (required by JSON-RPC 2.0 spec), and its contents are adapter-supplied and not validated by this layer — which is correct.

**5. Are there agent-blocking concerns hidden in the design?**

None that are blocking. The `isOp()` unexported sentinel means the `DecodeOp` switch is exhaustive over the wire contract and external types cannot accidentally satisfy `Op` — this is safe and correct. All error values carry context beyond their sentinel identity. The `CapabilityLevel` enum is closed and all values are named constants. The only subtle issue is that an agent acting on `adapter-timeout` has no `retryable` field in `ErrorData` — it must apply per-class retry policy from its own knowledge of what `adapter-timeout` means. Given the class set is small, this is acceptable for v1 (O3).
