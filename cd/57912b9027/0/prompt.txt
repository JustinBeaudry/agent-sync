**Unit 6 approach** (`conformance/echo/`):
- Single `main.go` that:
  1. Constructs an `adapterkit.Server` named `echo/v0.1` (or whatever shape the adapterkit Server constructor expects).
  2. Builds capabilities declaring every IR kind from `internal/ir.AllKinds()` as supported using `adapterkit.NewCapabilities()`.
  3. Declares one output: `{Path: ".echo", Mode: OutputModeOwnedSubdir}`.
  4. Registers an `OnEmit` handler that, for each IR node, emits one `mkdir` + one `write_file` under `.echo/<id>.md` with the node body as content.
  5. Calls `s.Run(ctx)`.
- The handler does NOT write files ŌĆö it returns ops in the EmitResult, and the runtime applies them. (AGENTS.md invariant: "Adapters never write files directly.")
- The README.md is a one-page "this is the canonical reference adapter; copy this directory as a starting point, change the manifest, change the OnEmit handler" guide.
- The test file (`echo_test.go`) builds the echo binary and runs the conformance harness against it. Asserts every happy-path corpus case passes (or skips appropriately for cases that test invariants the echo binary doesn't naturally trigger). Use the same `buildTestdataBinary`-style content-hashed cache pattern as `internal/adapter/subprocess_test.go`.
- The `conformance/echo/` directory is at the repo root (not under `internal/` or `cmd/`) so it ships as a real Go program in `go build ./...` and is reachable as a published reference.

**Unit 7 approach** (CLI subcommand):
- `internal/cli/cmd_adapter.go` follows the `cmd_trust.go` factory pattern strictly:
  - `AdapterDeps` struct with `Out io.Writer`, `Err io.Writer`, `In io.Reader`, `Now func() time.Time` (for deterministic JSON timestamps if any).
  - `NewAdapterCommand(deps)` returns the `adapter` parent.
  - `newAdapterConformanceTestCmd(deps)` returns the `conformance-test` leaf.
- Subcommand flags:
  - Positional arg: `binary` (required, validated to exist + be executable).
  - `--verbose, -v`: per-case detail (frames sent/received, ops emitted, expected vs actual diffs) on failure.
  - `--format=json|text`: output format (default `text` for human use; `json` for CI parsing).
  - `--filter=<regex>`: run only cases matching the pattern.
  - `--timeout=<duration>`: override per-case timeout (default: runtime defaults).
- Implementation:
  - Build `conformance.RunOptions` from the flags.
  - Call `conformance.Run(ctx, binary, opts)`.
  - Render the resulting `Report` per the chosen format.
  - Return an exit error sized to: 0 if all cases passed (skips don't count as failures), 1 if any failed, 2 if the binary itself couldn't be spawned.
- Output schemas:
  - Text: human-readable per-case `pass`/`fail`/`skip` lines + final summary table.
  - JSON: `{"cases": [{"name", "status", "reason"}], "summary": {"total", "passed", "failed", "skipped"}, "version": "aienvs/v1"}`.
- DO NOT modify `cmd/aienvs/main.go` ŌĆö Unit 16 of the parent plan owns root-level command wiring. Just make `NewAdapterCommand` independently testable via the same `testEnv` pattern as `cmd_trust_test.go`.
- Tests:
  - Happy path: `aienvs adapter conformance-test <built-echo-binary>` exits 0; text output matches expected per-case lines + summary.
  - JSON format: `--format=json` produces parseable JSON matching the documented schema.
  - Spawn error: `aienvs adapter conformance-test /does/not/exist` exits 2; error message clearly distinguishes "binary doesn't exist" from "binary exists but failed conformance".
  - All-fail: `aienvs adapter conformance-test <crashy-binary>` exits 1; per-case failures reported.
  - `--filter='happy-.*'` runs only happy-path cases.
  - `--timeout=1ms` causes every case to timeout; exits 1 with timeout-classified failures.

**Unit 8 approach** (spec freeze):
- Write `docs/spec/adapter-protocol-v1.md` with YAML frontmatter:
  ```
  ---
  title: aienvs Adapter Protocol v1
  status: frozen
  version: aienvs/v1
  date: 2026-04-26
  ---
  ```
- Section structure (final ordering at your discretion; the plan suggests):
  1. Overview ŌĆö what the protocol is, who it's for
  2. Framing ŌĆö LSP-style `Content-Length: <n>\r\n\r\n<body>`. `Content-Type` reserved; v1 ignores
  3. JSON-RPC envelope ŌĆö request/response/notification shapes; ID correlation; error object format including `data.error_class`
  4. Lifecycle ŌĆö `initialize` ŌåÆ `initialized` ŌåÆ `emit*` ŌåÆ `shutdown`. Magic-cookie handshake. Adapter MUST NOT emit ops before receiving `initialized`
  5. Capabilities object ŌĆö `concept_kinds` map (kind ŌåÆ `supported` | `partial` | `unsupported`), `write_tool_owned`, future-extension reservations
  6. Declared outputs ŌĆö `path` + `mode` (`owned-subdir` only in v1; `tool-owned` reserved). The integrity gate: any op outside `declared_outputs` is `adapter-undeclared-output`
  7. Op set ŌĆö `write_file`, `write_tool_owned`, `mkdir`, `delete`, `warning`. Field-by-field schema for each. Encoding rules (`utf8` vs `base64`; required when bytes are non-UTF-8)
  8. Errors ŌĆö JSON-RPC standard codes; `data.error_class` values: `adapter-panic`, `adapter-timeout`, `adapter-protocol-mismatch`, `adapter-undeclared-output`, `adapter-exec-denied`, `adapter-capability-lied`. Exit-code taxonomy (adapter `exit_code` + `stderr_tail`)
  9. Magic cookie ŌĆö `AIENVS_ADAPTER_COOKIE` env var, MUST echo verbatim in `initialize` result. Adapters that don't echo it fail with `adapter-exec-denied`
  10. `_meta` field ŌĆö reserved on every envelope; v1 ignores; future versions may carry implementation hints additively
  11. Timeouts ŌĆö handshake 5s, per-emit 30s, shutdown 5s. Defaults only; per-method overrides reserved for future
  12. Versioning policy ŌĆö Capabilities grow additively under `Capabilities{}`. `aienvs/v1` envelope/framing is frozen. `aienvs/v2` is reserved for envelope-level breaking changes
  13. Reserved for Unit 8b ŌĆö `$/cancelRequest`, `$/progress`, per-method timeouts, LSP error codes (`-32800`/`-32801`/`-32803`), version counter-proposal
  14. Examples ŌĆö canonical handshake, canonical emit, error response. Each example tagged with the directive convention below
  15. Conformance ŌĆö points at `internal/adapter/conformance/corpus/` and `aienvs adapter conformance-test`
