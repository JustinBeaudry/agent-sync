<approach>
Per-scope decision, placed where the loaded manifest is in hand (in or adjacent to prepareScope, and in the single-scope path). For EACH scope:

1. Eligibility: auto-advance applies only when the source is git-backed (Canonical.URL or Canonical.LocalPath) AND NOT `--frozen` AND `Canonical.Auto` is not explicitly false (nil ⇒ ON). `Canonical.LocalDir` sources are a NO-OP (no pin to advance). Frozen / auto:false / local_dir ⇒ today's behavior unchanged (do NOT acquire the CLI run lock for these; let engine.Sync self-lock as it does now).

2. For an eligible scope, acquire the scope's CLI-level RunLock (like update) and hold it across re-pin + prepare + sync. Then:
   a. Resolve newest with OFFLINE FALLBACK. For URL sources, use git.Materialize with `ResolveOrFallback: true`, `PinnedSHA: m.Canonical.Commit`, `Ref: m.Canonical.Ref`. If `Result.FellBackToPinned` (remote unreachable) ⇒ WARN, do NOT advance, proceed to prepare+sync at the existing pin. For LocalPath sources, resolve the ref locally (offline is not a concern).
   b. FAST-FORWARD BASELINE FIX (review finding #2): compute the ff proof against the SAME SHA the trust engine gates on — `m.TrustedSHA` (ManifestTrustedSHA) — NOT `m.Canonical.Commit`. If the baseline (trusted_sha) is empty, treat fastForward as FALSE (fail-safe), never default-true. Add/adjust an auto-path resolve in advance.go so the ff baseline is trusted_sha for the auto path. (Leave update's existing resolveAdvance using Canonical.Commit unchanged — update writes them equal and has a human gate.)
   c. Load trust State: open the trust Store and Fold for the canonical URL so revocations are seen. Call trust.Decide with Posture=PostureAllowNewSHAs, FastForward=<the ff verdict from (b)>, ManifestTrustedSHA=m.TrustedSHA, StateLoaded=true, and the resolved SHA.
   d. On `KindProceedAutoAdvance`: append the returned AppendPending to pending.jsonl (reuse the existing writer), repinManifest(newSHA), then continue to prepare+sync (which now materializes the new pin). On a refusal (KindDecisionRequired from non-ff, or ErrRevokedTrustAnchor, or any error): do NOT advance, report the scope (continue-and-report), keep the existing pin, and — for revoke — surface the revoke error clearly. NEVER prompt/hang in non-interactive mode.
   e. Re-pin MUST happen BEFORE prepareScope so prepare reads the advanced manifest and materializes the new SHA.
   f. If re-pin SUCCEEDS but the subsequent engine.Sync FAILS: surface a distinct condition/exit signal meaning "pin moved but files did not land — re-run sync" (model on cmd_update.go's exitUpdatePinMoved = 6). Do not roll back the pin silently.

3. --frozen threading (review finding #3): replace the context.WithValue bridge (withSyncRequest/syncRequestFrom) with an explicit field threaded through the options/request the prepare path consumes (e.g. on engine.Options or a prepare parameter). Both `--frozen` and `Canonical.Auto == false` must force the pinned-only path. Remove the now-dead context bridge if fully replaced.
